# 008 灵签信五个二级工具共用问卦排版

- 状态：已完成本地实现与浏览器验证；人物档案规则由决定 012 修订
- 日期：2026-07-24
- 关联 Issue：#19

## 决定

“云签解惑”“事业灵签”“流年运势”“时辰运势”“AI 解读报告”不建立五套独立视觉模板，统一复用“长期问卦”的完整表单排版：

```text
统一页眉、宣纸容器、章节标题、表单控件与唯一主操作
    ↓
根据工具增减必要输入
    ↓
直接进入该工具的差异化结果页
```

输入页保持相同的阅读节奏和组件骨架；差异集中在“填写什么”和“结果如何组织”，而不是重新发明导航、卡片、颜色或步骤。

## 工具层级

| 工具 | 必要输入 | 是否需要人物档案 | 结果页主层级 |
|---|---|---:|---|
| 云签解惑 | 人物档案、所问方向、一个具体问题 | 是 | 签象、本卦原文、动爻、当下提醒 |
| 事业灵签 | 人物档案、事业情境、一个具体选择或问题 | 是 | 当前结构、变化位置、变化后结构、决策检查 |
| 流年运势 | 所观年份、关注方向、具体问题、人物档案 | 是 | 年度背景、年度主象、关键变化、之卦、现实提醒 |
| 时辰运势 | 人物档案、所问时刻、所在地、一个具体问题 | 是 | 时间卦、时刻信息、本卦、动爻、之卦 |
| AI 解读报告 | 人物档案、一条已保存的问卦记录、解读重点 | 是 | 问卦摘要、卦象结构、原文依据、结构化提示 |

人物档案现统一作为五个工具的首个表单模块，用于人物识别与记录归属。除流年运势外，档案出生信息不参与其他工具既有起卦或报告算法。

人物档案的选择、新建、编辑、地区与经度逻辑实现为共享组件。“观星问卦”、流年运势以及后续任何需要出生信息的功能都使用同一组件和本地存储口径，不复制表单。

## 易学口径

- 云签、事业灵签和流年运势沿用项目已有的数字化四值起卦，结果必须展示本卦、动爻、之卦和已核对的《周易》原文。
- 时辰运势使用可重算的时间起卦口径：农历年支、月、日取上卦，再加所问时支取下卦与动爻；界面必须同时展示时区、所在地、农历日期、时辰和公式说明。
- AI 解读报告首期不伪装成已经接入外部大模型。未接入模型时明确标注“结构化解读原型”，只重组已保存卦象、原文与变化关系。
- 古籍原文、起卦数据和现代辅助解释分区呈现。现代提示不得冒充卦辞、爻辞或《象传》原文。
- 所有结果只用于观察变化与整理处境，不替代医疗、法律、投资或其他现实决策。

## 交互约束

- 五个首页入口均打开同一个全屏流程容器，并保留原入口上下文。
- 输入页不使用步骤条或连续“下一步”；一个页面完成必要信息，只有生成结果时发生页面状态切换。
- 每页只有一个清晰主操作。
- 返回修改信息时保留已填内容。
- 空状态必须真实：没有可供 AI 报告使用的问卦记录时，提示用户先完成一次问卦，不注入伪造历史记录。

## 视觉约束

- 复用 `reading-flow.css` 的米纸背景、宋/楷标题、茶棕正文、古金状态与控件。
- 桌面和移动端共享信息架构；移动端使用自然纵向滚动，不增加额外页面层级。
- 结果页可以改变内容顺序和分组，但不得脱离现有品牌体系。

## 实现落点

- `src/oracle-tool-flow.jsx`：五个工具共用的全屏输入与结果容器。
- `src/profile-archive-form.jsx`：人物档案共享组件，供观星问卦与流年运势复用。
- `src/reading-storage.js`：本地问卦记录存取，供 AI 解读报告读取真实记录。
- `src/iching.js`：时辰运势所需的可重算时间起卦逻辑。
- `design-qa.md`：390px 与 1440px 浏览器验证、交互与无障碍检查记录。
